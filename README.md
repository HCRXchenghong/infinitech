# Infinitech

一站式本地生活服务平台主仓。

这个仓库正在从“可运行的多端业务系统”持续整治为“可扩展、可压测、可回滚、可观测”的生产级平台。目标不是口头承诺“单机 10 万在线”，而是把核心链路改造成在高并发下不会因为单点设计直接崩掉的架构。

当前整治主范围固定为：
- `backend/go`
- `backend/bff`
- `socket-server`
- `admin-vue`
- `user-vue`
- `app-mobile`
- `merchant-app`
- `rider-app`

保留但不作为当前主整治范围：
- `android-user-app`
- `ios-user-app`
- `admin-app`

## 1. 平台定位

平台目标是做一个可持续扩展的一站式生活服务大平台，覆盖：
- 用户消费端
- 商家经营端
- 骑手履约端
- 管理后台
- 实时消息与客服
- 推送通知
- 财务与积分
- 营销与运营活动
- 地图、短信、微信登录等外部能力

当前业务模块已经包含外卖、跑腿、医药、公益、会员、同频饭友、消息客服、后台运营等，但仓库仍处于持续整治与主链路收口阶段。

## 2. 当前生产基线

当前已经明确的生产方向：
- 生产数据库：`PostgreSQL`
- 生产缓存与分布式状态：`Redis`
- `SQLite` 只允许本地开发、临时测试、历史兼容，不再作为生产主链路前提
- 根目录作为唯一主仓
- 每个整治批次都要求：代码改动、验证通过、README 更新、推送 GitHub

RTC 与首页推广当前规则也已经定死：
- `App / H5`：规划支持站内 1v1 音频通话
- `小程序`：不做音频通话，直接走系统电话
- 首页推广第一阶段：只做线下谈单 + 后台手工配置
- 同时支持商户位和商品位
- 不做竞价，不做按曝光/点击自动扣费，不做商户自助投放

## 3. 目录说明

核心业务目录：
- `backend/go`：Go 主 API 服务，逐步收口为业务事实源
- `backend/bff`：Node.js BFF 与统一代理层
- `socket-server`：实时网关与实时事件桥接
- `admin-vue`：管理后台 Web
- `user-vue`：用户消费端 uni-app
- `app-mobile`：用户 App uni-app
- `merchant-app`：商家端 uni-app
- `rider-app`：骑手端 uni-app

基础设施与共享目录：
- `shared/mobile-common`：多端共享移动端逻辑
- `backend/docker`：本地基础设施编排
- `backend/shared`：后端共享文件
- `backend/redis`：Redis 相关配置
- `scripts`：仓库级脚本
- `.github`：CI 与工作流

## 7. 已完成的关键整治

### 7.1 安全与鉴权
- 收紧了 Socket 聊天身份伪造与越权访问问题
- 收紧了后台鉴权与默认密钥回退问题
- 收紧了高风险删除操作与二次校验链路

### 7.2 配置治理
- 清理了大量开发机内网地址硬编码
- 真正会变化的运营数据、第三方服务参数逐步回收到后台配置
- 地图能力已整理为统一后端入口，后续可切换到天地图

### 7.3 主链路补齐
- 手机号改绑链路已打通
- 图形验证码链路已打通
- 邀请码注册与奖励链路已打通
- 用户资料编辑已接到服务端
- 骑手异常上报已打通
- 跑腿、医药等一批“假打通”页面已经改成真实接口

### 7.4 地址簿服务端化
- 用户端与 App 端地址列表、地址编辑、默认地址、确认下单已接入真实服务端地址
- 订单创建优先引用服务端地址对象，不再只信本地字符串

### 7.5 消息与客服统一
- 用户端、App 端、商家端、骑手端、后台工作台已经开始持续收口到服务端会话模型
- `/api/messages/conversations`、历史消息、已读同步已经接入主链路
- Socket 层已开始把消息和已读状态同步回 Go 服务
- 客服聊天主链已经不是单纯本地 storage 假会话

### 7.6 推送链路推进
- 四端设备注册、反注册与本地状态管理已接入
- `received / opened` 回执桥已接入
- 后台推送统计接口、投递明细接口已补齐
- 推送投递记录已能 materialize 成 delivery 明细
- Go 端已补后台 delivery 派发执行器，支持 `log` 与 `webhook` provider
- delivery 状态已经形成 `queued -> dispatching -> sent / retry_pending / failed -> acknowledged` 闭环

### 7.7 乱码与假数据清理
- 活跃页面中的大批假数据、假成功提示、本地伪状态已持续清理
- 多端活跃页面和关键维护文件中的可见乱码已持续修复
- 仓库 Markdown 已收口到当前这一份 `README.md`

### 7.8 本轮新增完成
- `socket-server/riderNamespace.js` 已开始把骑手直聊消息同步回 Go 权威消息服务，不再只落本地 `chat.db`
- 这让骑手直聊消息也进入统一的服务端会话/历史模型，为后续彻底收口 `chat.db` 继续往前推进了一步
- `admin-vue/src/views/HomeCampaigns.vue` 已重写成干净的首页推广管理页，后台可直接查看商户区 / 商品区的当前生效位次、来源和前台标识
- 首页推广服务补上了回归测试，已经覆盖“手工锁位优先于付费推广”和“活动有效状态回显”这两条核心规则
- `rider-app/App-logic.ts` 已把 `/rider` 命名空间收到的直聊消息转进统一前端实时事件流，骑手当前聊天页不再只能靠重新进页拉历史才能看到新消息
- `admin-vue/src/views/useChatConsole.js` 已在实时消息、已读回执后延迟回拉服务端会话列表，后台会话摘要与未读开始重新锚回服务端权威源

## 8. 当前仍未完成的大项

### 8.1 生产架构硬化
- Go 主库仍需彻底切到 `PostgreSQL` 默认生产路径
- Redis 在生产环境下仍需改成强依赖并 fail-fast
- BFF / Go / Socket 的统一限流、超时、请求体限制、标准错误模型还要继续补齐
- 上传链路还需要进一步从本地/内存模式收口到流式存储抽象
- 健康检查、readiness、结构化日志、请求 ID、关键业务埋点仍要继续补齐

### 8.2 消息系统彻底统一
- `socket-server/chat.db` 还没有完全退出“业务事实源”角色
- 商家端、骑手端、后台工作台剩余本地伪会话和伪未读还要继续清掉
- 多端消息列表、历史、未读、已读仍需继续统一成一套权威口径

### 8.3 推送系统闭环
- 当前已完成设备登记、回执、统计、delivery materialization、后台派发执行器
- 当前仍未完成：
  - 真正的外部厂商推送网关接入
  - 更细粒度的终端送达展示
  - 更完整的失败原因归类与重试观测面板

### 8.4 首页商户 / 商品推广位
- 规则已经确定，但后台能力尚未正式落地完成
- 第一阶段继续采用：
  - 管理员手工锁定位次优先
  - 已审核且有效的付费推广计划次之
  - 普通今日推荐再次之
  - 自然排序最后
- 同时支持商户位和商品位
- 前台统一显示 `推广` 标识

### 8.5 P2P 音频通话
- `App / H5` 规划支持站内 1v1 音频通话
- `小程序` 明确不做音频通话，直接显示系统电话
- 当前仍待落地：
  - 服务端信令
  - 会话/订单关系鉴权
  - 通话元数据记录
  - 投诉与录音保留策略
  - 后台通话记录与投诉处理

## 9. 首页推广位方案

首页推荐不做复杂竞价信息流，第一阶段继续保留并升级现有两条能力：
- 商户“今日推荐 / 位次”
- 商品“今日推荐 / 位次”

排序优先级固定为：
1. 管理员手工锁定位次
2. 已审核且有效的付费推广计划
3. 普通今日推荐
4. 自然排序

第一阶段定向能力只做：
- 全局
- 可选城市
- 可选业务分类

明确不做：
- 用户画像定向
- 自动竞价
- 商户自助投放

## 10. 配置治理原则

继续遵守这个边界：
- 真正会变化的运营数据放后台配置
- 外部依赖参数放后台配置
